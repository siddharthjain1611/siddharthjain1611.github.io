Context

The Setting

An enterprise customer support system powered by an LLM-based chat agent. The agent handles refund inquiries, warranty claims, and escalation routing. It operates under a set of business policies: refund limits, escalation thresholds, regulatory requirements.

The standard approach to enforcing these policies is to include rules in the system prompt — often growing to 4,000+ tokens of instructions.

Problem

Prompt-Stuffing Fails at Scale

As system prompts grow, models exhibit attention degradation: effective attention to any single rule decreases as more rules are added. Instructions buried in the middle of a long prompt are probabilistically forgotten.

The result: an agent that confidently promises a $200 refund that violates policy, then the human team reverses it — destroying trust in both directions. In the first week of deployment, 17% of conversations contained at least one policy violation (refund commitments: 61%, warranty interpretations: 28%).

Safe AI is not a prompting problem. It's an architectural problem. The policy layer should be as separate from the language layer as a database is from an API.

Solution

Parallel Shadow-Evaluator via MCP Server

The key insight: the Primary Agent and Evaluator Agent don't need to run sequentially. They can run in parallel on the same input context, with a gate that holds the response until the evaluator clears it.

The architecture separates three concerns:

  • Primary Agent: A fast, lightweight model optimized for tone and conversational fluency. Has no awareness of policy rules — only generates human-sounding responses.
  • Evaluator Agent (Shadow): A specialized model with strict JSON-schema output. Runs in parallel on the same context. Has access to the company's deterministic Rules Engine API for refund limits, escalation thresholds, and regulatory checks. Returns structured confidence scores per policy category.
  • MCP Intercept Gate: Before the Primary Agent's response reaches the websocket, the orchestrator checks the Evaluator's aggregate confidence score. Below 0.95 threshold → response blocked, replaced by pre-approved fallback or routed to human agent.

Key Insight

Why This Worked

Separation of concerns at the agent level. The Primary Agent doesn't need to know about policy — it only generates fluent language. The Evaluator doesn't need to be fluent — it only needs to be right. Specialization made both fast and accurate.

This architecture became the template for every new AI feature deployment: before an agent goes to production, it gets a bespoke Evaluator Agent with domain-specific policy schemas. Safety is architectural, not prompt-level.
