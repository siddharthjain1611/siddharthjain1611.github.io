01 / The Problem

Prompt-Stuffing Fails. Always.

The industry's default response to LLM hallucinations in enterprise support is to add more rules to the system prompt: "Never promise refunds over $50. Do not discuss competitor pricing. Always escalate tier-2 complaints."

At small scale, this works. At production scale, it fails for a fundamental reason: attention degradation. As the system prompt grows, the model's effective attention to any single rule decreases. By the time your rules document hits 4,000 tokens, the model is probabilistically forgetting instructions buried in the middle.

The result: a support agent that confidently promises a customer a $200 refund that violates company policy, then apologizes when the human team reverses it — destroying trust in both directions.

"Safe AI is not a prompting problem. It's an architectural problem. The policy layer should be as separate from the language layer as a database is from an API."

02 / The Hacky V1

Post-Hoc Asynchronous Auditing

Our first attempt at safety was asynchronous. We piped all LLM-generated chat transcripts into a nightly batch job. A smaller, cheaper model flagged conversations that appeared to contain policy violations. A human team reviewed the flags the next morning.

This was excellent for analytics — we could precisely quantify the violation rate (17% of conversations in the first week) and identify which policy categories were most frequently violated (refund commitments: 61%, warranty interpretations: 28%).

But for the customer, the damage was already done. The violation had been made, the expectation set, the trust broken — or worse, the incorrect promise acted upon. Post-hoc auditing was a measurement tool, not a prevention tool.

03 / The Wall

04 / The Real Solution

The Parallel Shadow-Evaluator via MCP Server

Instead of a linear chain, we rebuilt the architecture around a Model Context Protocol (MCP) server acting as an orchestration layer. The key insight: the Primary Agent and Evaluator Agent don't need to run sequentially — they can run in parallel on the same input context.

The architecture had three key components:

  • The Primary Agent: A fast, lightweight model (optimized for tone and conversational fluency) generates the draft response. It has no awareness of policy rules — it only knows how to sound human and helpful.
  • The Evaluator Agent (Shadow): A specialized model with strict JSON-schema output runs in parallel on the same conversation context. It has access to the company's internal Rules Engine API — a deterministic service that enforces refund limits, escalation thresholds, and regulatory requirements. It returns a structured confidence score for each policy category.
  • The MCP Intercept Gate: Before the Primary Agent's response hits the websocket, the Orchestrator checks whether the Evaluator's aggregate confidence score meets the 0.95 threshold. Below threshold → the response is blocked and replaced by a pre-approved fallback template or routed to a human agent.

The core insight that made this work: separating the generation concern from the safety concern. The Primary Agent doesn't need to know about policy — it just needs to generate fluent language. The Evaluator doesn't need to be fluent — it just needs to be right. Specialization at the agent level is what made both fast and accurate.

This architecture is now the template for every new AI feature deployment. Before a new agent goes to production, it gets a bespoke Evaluator Agent with domain-specific policy schemas.
