Policy Enforcement

The Situation

An LLM-powered support agent with business rules

An enterprise support chatbot handles refund inquiries, warranty claims, and escalation routing. It operates under a set of business policies — refund limits, escalation thresholds, regulatory requirements. These rules are currently stuffed into the system prompt (~4,000 tokens of instructions).

The Reframe

This isn't a prompting problem. It's a separation-of-concerns problem.

The default approach: add more rules to the system prompt. "Never promise refunds over $50. Always escalate tier-2 complaints."

This fails at scale due to attention degradation — as the prompt grows, the model's effective attention to any single rule decreases. Instructions buried in the middle are probabilistically forgotten. The result: an agent that confidently makes promises that violate policy.

The real problem: asking a language model to simultaneously be fluent AND be compliant is asking one system to optimize for two conflicting objectives. Fluency and policy compliance are separate concerns and should live in separate components.

The policy layer should be as separate from the language layer as a database is from an API. You wouldn't put validation logic inside the response serializer.

Why Option D: The Primary and Evaluator don't have a data dependency — they both operate on the same input (user message + conversation context). Running them in parallel means total latency ≈ max(primary, evaluator) rather than sum(primary, evaluator).

The Bet

Parallel shadow-evaluator via MCP orchestration

The bet: separate generation from compliance entirely. The Primary Agent only optimizes for fluency. The Evaluator only optimizes for correctness. An orchestration layer (MCP server) runs both in parallel and gates the response on the Evaluator's score.

Three components, three separate concerns:

  • Primary Agent: Lightweight model optimized for conversational fluency. Has zero knowledge of policy rules. Only job: sound helpful and human.
  • Evaluator Agent: Specialized model with strict JSON output schema. Calls a deterministic Rules Engine API for refund limits, escalation thresholds, regulatory checks. Returns confidence scores per policy category.
  • Intercept Gate: Holds the Primary's response in buffer. If Evaluator's aggregate score ≥ 0.95, releases to user. If below, blocks and substitutes a pre-approved fallback or routes to human.

Key Tradeoffs

The hard calls

  • Parallel execution vs streaming-then-gating: An alternative is to stream the Primary's response token-by-token and evaluate on-the-fly. This gives lower perceived latency (user sees typing indicator immediately). But partial responses that get blocked mid-sentence create a worse UX than a slightly delayed complete response. Chose parallel with full-response gating.
  • 2x compute cost vs single-model safety: Running two models costs roughly double the inference budget. But the alternative (prompt-stuffing) has a ~40% miss rate at scale. The cost of one policy violation reaching a customer (reversal cost + trust damage) far exceeds the marginal compute cost.
  • 0.95 threshold vs lower/higher: Lower threshold (0.90) lets more borderline responses through — lower false positive rate but higher violation risk. Higher threshold (0.99) blocks too aggressively — good responses get caught and users see too many fallbacks. 0.95 is calibrated to the domain's risk tolerance.
Open Questions

What could go wrong

Evaluator drift: If the Rules Engine API changes (new policy added, threshold updated) but the Evaluator's training data doesn't reflect this, the Evaluator might not catch new violation types. Need a feedback loop where blocked/unblocked decisions are periodically audited against current policy state.

Adversarial users: What happens when a user deliberately tries to elicit policy violations — e.g., framing a request so the Primary is likely to comply and the violation is subtle enough to pass the Evaluator? Need red-team testing specifically targeting the parallel architecture's blind spots.

Fallback fatigue: If the block rate is too high, users repeatedly hit generic fallback responses and lose trust in the system's competence. The fallback itself needs to feel helpful, not robotic — perhaps routing to human immediately rather than a canned response.
